708 * # UI declarative data binding {#UI_declarative_data_binding}
709 * As said earlier, \c let syntax is a little bit clunky and requires extra boilerplate code to set up.
710 *
711 * Here's where declarative syntax comes into play. The logic behind the syntax is the same as in
712 * `AObject::connect`/`AObject::biConnect` (for ease of replacement/understanding).
713 *
714 * Declarative syntax uses `&` and `&&` operators to set up connections. These were chosen intentionally: `&&` resembles
715 * chain, so we "chaining view and property up".
716 *
717 * - `&` sets up one-directional connection (`AObject::connect`).
718 * - `&&` sets up bidirectional connection (`AObject::biConnect`).
719 *
720 * Also, `>` operator (resembles arrow) is used to specify the destination slot.
721 *
722 * The example below is essentially the same as @ref "Label_via_let" but uses declarative connection set up syntax.
723 *
724 * ## Label via declarative {#Label_via_declarative}
725 * Use `&` and `>` expression to connect the model's username property to the label's @ref ALabel::text "text"
726 * property.
727 * @code{cpp}
728 * using namespace declarative;
729 * struct User {
730 * AProperty<AString> name;
731 * };
732 *
733 * auto user = aui::ptr::manage(new User { .name = "Roza" });
734 *
735 * class MyWindow: public AWindow {
736 * public:
737 * MyWindow(const _<User>& user) {
738 * setContents(Centered {
739 * _new<ALabel>() & user->name > &ALabel::text
740 * });
741 * }
742 * };
743 * auto window = _new<MyWindow>(user);
744 * window->show();
745 * @endcode
746 *
747 * ![text](imgs/UIDataBindingTest.Label_via_declarative_1.png)
748 * Note that the label already displays the value stored in User.
749 *
750 * Let's change the name:
751 * @code{cpp}
752 * user->name = "Vasil";
753 * @endcode
754 *
755 * ![text](imgs/UIDataBindingTest.Label_via_declarative_2.png)
756 * In this example, we've achieved the same intuitive behaviour of data binding of `user->name` (like in
757 * @ref "Label_via_let" example) but using declarative syntax. The logic behind `&` is almost the same as with `let`
758 * and `AObject::connect` so projection use cases can be adapted in a similar manner.
759 *
760 * ## ADataBindingDefault for omitting view property {#ADataBindingDefault_for_omitting_view_property}
761 * In previous example we have explicitly specified ALabel's property to connect with.
762 *
763 * One of notable features of declarative way (in comparison to procedural `let` way) is that we can omit the view's
764 * property to connect with if such `ADataBindingDefault` specialization exist for the target view and the property
765 * type. Some views have already predefined such specialization for their underlying types. For instance, ALabel has
766 * such specialization:
767 *
768 * @code{cpp}
769 * /* PREDEFINED! You don't need to define it! This listing is an example */
770 * template<>
771 * struct ADataBindingDefault<ALabel, AString> {
772 * public:
773 * static auto property(const _<ALabel>& view) { return view->text(); }
774 * };
775 * @endcode
776 *
777 * We can use this predefined specialization to omit the destination property:
778 * @code{cpp}
779 * _new<ALabel>() & user->name
780 * @endcode
781 *
782 *
783 * Behaviour of such connection is equal to @ref "Label_via_declarative":
784 *
785 * ![text](imgs/UIDataBindingTest.Label_via_declarative_1.png)
786 * Note that the label already displays the value stored in User.
787 *
788 * Let's change the name:
789 * @code{cpp}
790 * user->name = "Vasil";
791 * @endcode
792 *
793 * ![text](imgs/UIDataBindingTest.Label_via_declarative_2.png)
794 * In this example, we've omitted the destination property of the connection while maintaining the same behaviour
795 * as in @ref "Label_via_declarative".

797 * ## ADataBindingDefault strong type propagation {#ADataBindingDefault_strong_type_propagation}
798 * Think of `ADataBindingDefault` as we're not only connecting properties to properties, but also creating a
799 * "property to view" relationship. This philosophy covers the following scenario.
800 *
801 * In AUI, there's aui::ranged_number template which stores valid value range right inside the type:
802 * @code{cpp}
803 * struct User {
804 * AProperty<aui::ranged_number<int, 1, 99>> age;
805 * };
806 * @endcode
807 *
808 *
809 * These strong types can be used to propagate their traits on views, i.e., ANumberPicker. When using declarative
810 * syntax, the property system calls `ADataBindingDefault::setup` to apply some extra traits of the bound value on
811 * the view. Here's an abstract on how `ANumberPicker` defines specialization of `ADataBingingDefault` with
812 * `aui::ranged_number`:
813 * @code{cpp}
814 * /* PREDEFINED! You don't need to define it! This listing is an example */
815 * template <aui::arithmetic UnderlyingType, auto min, auto max>
816 * struct ADataBindingDefault<ANumberPicker, aui::ranged_number<UnderlyingType, min, max>> {
817 * public:
818 * static auto property(const _<ANumberPicker>& view) {
819 * return view->value();
820 * }
821 * static void setup(const _<ANumberPicker>& view) {
822 * view->setMin(aui::ranged_number<UnderlyingType, min, max>::MIN);
823 * view->setMax(aui::ranged_number<UnderlyingType, min, max>::MAX);
824 * }
825 * // ...
826 * };
827 * @endcode
828 *
829 * As you can see, this specialization pulls the min and max values from `aui::ranged_number` type and sets them
830 * to `ANumberPicker`. This way `ANumberPicker` finds out the valid range of values by simply being bound to value
831 * that has constraints encoded inside its type.
832 * @code{cpp}
833 * _new<ANumberPicker>() && user->age,
834 * @endcode
835 *
836 * @note
837 * We're using `operator&&` here to set up bidirectional connection. For more info, go to
838 * @ref "Declarative_bidirectional_connection".
839 *
840 *
841 * By creating this connection, we've done a little bit more. We've set ANumberPicker::setMin and
842 * ANumberPicker::setMax as well:
843 * @code{cpp}
844 * EXPECT_EQ(numberPicker->getMin(), 1);
845 * EXPECT_EQ(numberPicker->getMax(), 99);
846 * @endcode
847 *
848 *
849 * This example demonstrates how to use declarative binding to propagate strong types. `aui::ranged_number`
850 * propagates its constraints on `ANumberPicker` thanks to `ADataBindingDefault` specialization.

889 * ## Declarative bidirectional connection {#Declarative_bidirectional_connection}
890 * In previous examples, we've used `&` to make one directional (one sided) connection. This is
891 * perfectly enough for ALabel because it cannot be changed by user.
892 *
893 * In some cases, you might want to use property-to-property as it's bidirectional. It's used for populating view
894 * from model and obtaining data from view back to the model.
895 *
896 * For this example, let's use ATextField instead of ALabel as it's an editable view. In this case, we'd want to use
897 * `&&` because we do want `user->name` to be aware of changes of the view.
898 * @code{cpp}
899 * _new<ATextField>() && user->name
900 * @endcode
901 *
902 *
903 * This gives the following result:
904 * ![text](imgs/UIDataBindingTest.Declarative_bidirectional_connection_1.png)
905 *
906 * Let's change the name programmatically:
907 * @code{cpp}
908 * user->name = "Vasil";
909 * @endcode
910 *
911 *
912 * ATextField will respond:
913 * ![text](imgs/UIDataBindingTest.Declarative_bidirectional_connection_2.png)
914 *
915 * If the user changes the value from UI, these changes will reflect on `user->model` as well:
916 * ![text](imgs/UIDataBindingTest.Declarative_bidirectional_connection_3.png)
917 * @code{cpp}
918 * EXPECT_EQ(user->name, "Changed from UI");
919 * @endcode
920 *
921 *
922 * This way we've set up bidirectional projection via `&&` which makes `user->name` aware of UI
